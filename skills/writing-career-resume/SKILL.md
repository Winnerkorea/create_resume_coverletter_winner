# Writing Career Resume

## Attribution

이 스킬은 [yscho03/writing-career-resume-skill](https://github.com/yscho03/writing-career-resume-skill)의 한국어 기술 경력서 작성 Skill을 바탕으로 작성했다. 원본은 MIT License이며, 이 저장소에서는 Excel/Google Sheets 경력 입력, 채용공고 매칭률 분석, AI 보완 질문, `resume.md`/`coverletter.md` 생성, HTML/PDF 렌더링 흐름에 맞게 확장한다.

## Workflow

1. 사용자 입력에서 회사, 프로젝트, 기간, 담당업무, 문제 해결, 역할, 배운점, 기술, 학력, 자격증, 연락처, 지원 직무, 채용공고 요건을 추출한다.
2. 사용자가 양식을 지정하지 않으면 기본값으로 `resume_template: ats-basic`, `cover_letter_template: job-fit`을 사용한다.
3. 지원 회사의 자격요건/우대사항이 있으면 관련 경험과 기술을 우선 배치한다.
4. 경력과 회사 내 프로젝트를 최신순 내림차순으로 정렬한다.
5. 기간은 `YYYY. MM ~ YYYY. MM (N개월)` 또는 `YYYY. MM ~ 현재 (N년 N개월)` 형식으로 쓴다.
6. 담당업무는 수행한 업무 범위를 쓰고, 역할은 어떤 업무를 어떻게 담당했는지 상세하게 쓴다.
7. 배운점은 프로젝트를 통해 구체적으로 무엇을 배웠는지 쓴다.
8. 최종 결과는 HTML/PDF로 렌더링 가능한 Markdown만 출력한다. 사용자가 파일 생성을 요청하면 `resume.md`와 필요 시 `coverletter.md`로 저장한다.

정보가 부족하면 합리적으로 보완하되, 추정한 내용은 문서 밖의 짧은 메모로 표시하거나 사용자에게 확인한다. 경력 기간, 회사명, 직책처럼 사실성이 중요한 정보는 임의로 꾸미지 않는다.

## Source Of Truth

Markdown 문서를 원본 데이터로 취급한다. HTML과 PDF는 이 Markdown을 렌더링한 결과물이다.

- 이력서 내용은 HTML 구조에 맞추어 다시 만들지 않는다.
- 양식 선택은 Markdown의 핵심 내용을 바꾸는 기능이 아니라, 렌더링 방향과 강조 순서를 고르는 기능이다.
- Markdown은 제목, 표, 목록, 문단처럼 일반 Markdown 파서가 HTML로 변환하기 쉬운 문법을 사용한다.
- 복잡한 레이아웃 지시, 인라인 스타일, HTML 태그 남용을 피한다.
- 표 안의 줄바꿈은 Markdown 표 안정성을 위해 `<br>`만 허용한다.

권장 처리 흐름:

```text
사용자 경력 자료 + 채용공고
        ↓
resume.md / coverletter.md
        ↓
Markdown parser
        ↓
HTML preview
        ↓
선택한 CSS template
        ↓
PDF 출력
```

## Spreadsheet Input Schema

사용자가 Excel 또는 Google Sheets로 경력 자료를 정리하려는 경우 아래 컬럼을 기본 입력 양식으로 사용한다.

| NO | 업체 | Project | ISSUE | 담당 업무 | 문제 해결 | 배운점 |
| --- | --- | --- | --- | --- | --- | --- |
| 1 | 회사명 또는 고객사 | 프로젝트명 | 해결해야 했던 문제/상황 | 맡은 업무 범위 | 어떤 방식으로 해결했는지 | 구체적으로 배운 내용 |

컬럼 작성 기준:

- `NO`: 정렬과 참조를 위한 번호. 최신순 정렬 전 원본 순서를 보존하는 값으로 사용한다.
- `업체`: 재직 회사, 고객사, 발주처, 서비스 운영 주체 중 문맥에 맞는 이름을 쓴다.
- `Project`: 프로젝트명 또는 제품/서비스명을 쓴다.
- `ISSUE`: 당시 해결해야 했던 문제, 요구사항, 장애, 개선 과제를 쓴다.
- `담당 업무`: 본인이 맡은 업무 범위를 쓴다.
- `문제 해결`: 어떤 접근, 기술, 협업 방식으로 문제를 해결했는지 쓴다.
- `배운점`: 프로젝트를 통해 구체적으로 무엇을 배웠는지 쓴다.

Spreadsheet를 이력서 MD로 변환할 때는 다음처럼 매핑한다.

- `업체` → 회사명 또는 프로젝트 맥락
- `Project` → 경력 섹션의 프로젝트 제목 또는 담당업무 보조 정보
- `ISSUE` → 역할 설명의 문제 상황
- `담당 업무` → `담당업무`
- `문제 해결` → `역할`의 상세 구현/해결 내용
- `배운점` → `배운점`

기간, 기술 스택, 직무, 연락처가 Spreadsheet에 없으면 사용자 입력 또는 별도 메모에서 보완한다. 없으면 임의로 꾸미지 말고 확인이 필요한 항목으로 남긴다.

## Template Selection

사용자가 이력서 양식을 고를 수 있도록 문서 상단에 짧은 메타 정보를 포함한다. 렌더링 앱은 이 값을 읽어 CSS 양식을 선택할 수 있다.

```markdown
<!--
resume_template: ats-basic
cover_letter_template: job-fit
target_role: Frontend Engineer
target_company: 회사명
-->
```

이력서 양식은 다음 3개 중 하나를 사용한다.

- `ats-basic`: 채용 시스템과 심사자가 읽기 쉬운 단일 컬럼 기본형. 신입, 주니어, 대기업, 플랫폼 지원에 적합.
- `career-detail`: 프로젝트별 담당업무, 역할, 배운점을 자세히 보여주는 경력 강조형. 경력직, 프리랜서, SI/솔루션, 프로젝트 중심 지원에 적합.
- `portfolio-open-source`: 기술 스택, GitHub, 블로그, 오픈소스, 대표 프로젝트를 강조하는 포트폴리오형. 스타트업, 개발자, 오픈소스 기반 지원에 적합.

커버레터 양식은 다음 3개 중 하나를 사용한다.

- `job-fit`: 채용공고 요구사항과 내 경험을 직접 연결하는 직무 적합형.
- `growth-story`: 문제 해결 과정, 배운점, 커리어 방향성을 강조하는 성장 서사형.
- `impact-first`: 핵심 기여 가능성과 결과를 먼저 전달하는 임팩트 강조형.

양식이 선택되어도 경력 사실은 바꾸지 않는다. 선택한 양식에 따라 섹션 순서, 강조 문장, 커버레터 전개만 조정한다.

## Writing Rules

- 개조식으로 작성한다.
- `~하였습니다`, `~했습니다`, `~입니다`, `~개발했습니다`를 피한다.
- `~하였음`, `~처리함`, `~임`, `~개발`처럼 간결하게 쓴다.
- 불필요한 부사와 수식어를 제거한다.
- 프로젝트는 모두 나열하지 말고 임팩트 있고 지원 직무와 관련 높은 것 위주로 선별한다.
- 기술 스택은 부각되는 기술 순으로 정렬한다: 주력 언어, 주력 프레임워크, 데이터베이스, 인프라/도구, 기타.
- 채용공고가 있으면 요구 기술과 관련 프로젝트를 상위에 배치하고, 관련 기술을 먼저 표기한다.
- MD를 HTML로 보여줄 수 있도록 섹션 제목을 일관되게 유지한다.
- 커버레터는 이력서와 같은 사실을 바탕으로 쓰되, 이력서 표를 반복하지 않는다.

## Duty Role And Learning

`역활`이라고 입력된 경우에도 문서에는 `역할`로 정리한다.

담당업무는 맡은 업무 영역을 간결하게 설명한다.

- 챗봇 서비스 백엔드 API 개발
- React 기반 고객 대시보드 개발 및 운영
- 결제 장애 모니터링과 대응 프로세스 개선

역할은 어떤 업무를 하였고, 무엇을 어떻게 담당했는지 상세하게 설명한다.

- 실시간 응답 처리를 위해 SSE 스트림 처리 구조 설계 및 구현
- VectorDB 검색 품질 개선을 위해 Collection, Index, Metadata 구조 구성
- 장애 재현 시간을 줄이기 위해 결제 로그 수집 기준과 알림 조건 정리
- Spreadsheet의 `ISSUE`와 `문제 해결`을 연결해 문제 상황과 해결 방식을 함께 설명

배운점은 구체적으로 어떤 것을 배웠는지 설명한다.

- SSE 연결 유지와 예외 처리 방식에 따라 사용자 체감 응답성이 달라짐을 학습
- VectorDB 검색 품질은 임베딩 모델뿐 아니라 메타데이터 설계와 인덱스 전략의 영향을 받음을 학습
- 장애 대응 자동화는 알림 수보다 원인 파악에 필요한 로그 맥락을 줄이는 것이 중요함을 학습

## Cover Letter Template

커버레터가 필요한 경우 `coverletter.md` 형식으로 별도 작성한다.

```markdown
<!--
cover_letter_template: job-fit
target_role: 지원 직무
target_company: 지원 회사
-->

# 커버레터

안녕하세요. [지원 회사]의 [지원 직무]에 지원하는 [이름]입니다.

[채용공고의 핵심 요구사항]과 제 경험 중 [연결되는 프로젝트/역할]이 맞닿아 있다고 판단함.

[구체적 역할]을 수행하며 [담당업무]를 맡았고, 그 과정에서 [배운점]을 확인함.

입사 후에는 [기술/업무 강점]을 바탕으로 [회사 또는 팀에 기여할 방향]에 기여하고자 함.
```

커버레터 양식별 작성 기준:

- `job-fit`: 채용공고 요구사항 → 관련 경험 → 기여 방향 순서로 작성.
- `growth-story`: 문제 상황 → 맡은 역할 → 배운점 → 다음 기여 방향 순서로 작성.
- `impact-first`: 가장 강한 강점 또는 결과 → 근거 경험 → 회사 적용 가능성 순서로 작성.

## Period Rules

- 기간은 반드시 총 기간을 괄호로 붙인다.
- 기간 미지정 시 최대 1년 이내에서 자연스러운 기간을 제안하고, 추정임을 알린다.
- 너무 짧은 기간은 빠르게 종료된 인상을 줄 수 있으므로 담당업무와 역할이 분명할 때만 사용한다.
- 너무 긴 기간은 느슨한 프로젝트로 보일 수 있으므로 주요 마일스톤을 분리한다.

## Quality Checklist

문서 작성 후 다음을 점검한다.

- 문서 상단에 선택한 `resume_template`과 `cover_letter_template`이 있음
- 경력이 최신순으로 정렬됨
- 모든 기간에 `(N개월/N년)` 표기가 있음
- Excel/Google Sheets 입력이 있으면 `NO`, `업체`, `Project`, `ISSUE`, `담당 업무`, `문제 해결`, `배운점` 컬럼을 반영함
- 담당업무가 맡은 업무 범위를 명확히 설명함
- 문제 해결 내용이 역할에 구체적으로 반영됨
- 역할이 어떤 업무를 하였고 어떻게 담당했는지 상세함
- 배운점이 구체적으로 어떤 것을 배웠는지 설명함
- 개조식 문체만 사용함
- 불필요한 부사와 수식어를 제거함
- 기술 스택이 중요도순으로 정렬됨
- 핵심 프로젝트만 선별됨
- 지원 회사 요건이 있으면 관련 경험을 강조함
- Markdown이 HTML 렌더링 가능한 구조임
- 커버레터가 이력서와 같은 사실을 바탕으로 하되 단순 반복이 아님

## Example

```markdown
<!--
resume_template: career-detail
cover_letter_template: growth-story
target_role: Backend Engineer
target_company: Example Labs
-->

# 홍길동

hong@example.com | Seoul | GitHub: github.com/example

# 요약

- Python/FastAPI 기반 백엔드 API와 실시간 응답 처리 경험 보유
- VectorDB 검색 구조와 Kubernetes 배포 흐름을 프로젝트에서 직접 설계
- 문제 해결 과정에서 배운 내용을 문서화하고 팀 운영 기준으로 정리

# 경력

### (주) 예시회사

*Seoul - (2025. 03 - 2025.10)*
```
